# 财报 PDF 下载工具开发规范

## 1. 规范目标

本规范用于约束“财报 PDF 下载工具”的日常开发、代码提交、接口设计、UI 实现、测试和交付。所有实现必须优先满足三个原则：

1. 数据来源友好：批量下载必须保护巨潮资讯、AKShare 相关源站，不能暴力请求。
2. 用户结果可靠：宁可慢一点，也不能静默失败、下载错报告或覆盖用户文件。
3. 代码可维护：数据源、匹配规则、下载队列、UI 状态必须边界清楚，便于后续替换和扩展。
4. 测试优先：改动功能前先写测试用例，先看到测试失败，再实现功能并让测试通过。
5. 跨平台可用：默认支持 macOS 和 Windows 本地运行，代码不得绑定单一操作系统路径或启动方式。

配套需求文档见：[财报PDF下载工具开发文档.md](财报PDF下载工具开发文档.md)。

## 2. 技术栈约定

正式版本采用：

- 后端：Python 3.11+、FastAPI、Pydantic、requests/httpx、SQLite。
- 前端：React、TypeScript、Vite、lucide-react。
- 数据源：AKShare + 巨潮资讯底层公开接口。
- 本地存储：SQLite 记录任务、报告候选、下载结果、日志索引。

MVP 可以先不做桌面打包，但代码结构要预留后续 Tauri/Electron 或启动器包装空间。

## 3. 项目结构规范

推荐目录：

```text
backend/
  app/
    main.py
    config.py
    models.py
    services/
      cninfo_client.py
      akshare_client.py
      report_matcher.py
      rate_limiter.py
      downloader.py
      task_queue.py
      excel_importer.py
    storage/
      database.py
      repositories.py
    tests/
frontend/
  src/
    components/
    api/
    styles/
docs/
```

约束：

- 数据源请求只能放在 `cninfo_client.py`、`akshare_client.py` 或其明确子模块中。
- 报告标题匹配只能放在 `report_matcher.py` 中，前端不得复制匹配逻辑。
- 下载限速只能由 `rate_limiter.py` 统一控制，不允许业务代码自行 `sleep` 后直接请求源站。
- 文件保存与命名只能由 `downloader.py` 负责。
- UI 组件只负责展示和发起用户意图，不负责推导 PDF 地址。

## 4. 源站保护规范

这是硬性规范，不能为了“更快”绕过。

### 4.1 默认参数

| 项目 | 默认值 | 后端硬限制 |
| --- | --- | --- |
| 请求间隔 | `2.0 秒` | 不得低于 `1.0 秒` |
| 批量并发 | `1` | 不得超过 `3` |
| 请求超时 | `20 秒` | 可配置 |
| 单请求重试 | `3 次` | 不得无限重试 |
| 最大退避 | `60 秒` | 可配置 |

### 4.2 限速器要求

所有访问以下域名的请求必须经过全局限速器：

- `www.cninfo.com.cn`
- `static.cninfo.com.cn`
- AKShare 内部调用触发的巨潮请求

实现要求：

- 限速器按域名维护请求节奏。
- 公告查询、股票字典查询、PDF 下载都计入限速。
- 并发数大于 1 时，同一域名请求仍必须排队。
- 任务开始时记录本次限速参数到日志。
- 日志中可低频显示等待信息，避免刷屏。

### 4.3 自动降速

出现以下情况时必须自动降速：

- HTTP `429`
- HTTP `403`
- HTTP `5xx`
- 连续超时
- 连接重置
- 源站返回异常空响应

降速规则：

- 当前请求间隔乘以 `2`。
- 最大不超过 `60 秒`。
- 连续成功若干次后逐步恢复到用户设置值。
- 自动降速必须写入任务日志，如“检测到源站繁忙，请求间隔调整为 8.0 秒”。

### 4.4 禁止行为

- 禁止无间隔循环请求。
- 禁止失败后立即密集重试。
- 禁止为提升速度绕过后端限速器。
- 禁止默认高并发批量下载。
- 禁止隐藏源站限流错误。

## 5. 后端代码规范

### 5.1 类型与模型

- API 入参和出参必须使用 Pydantic Model。
- 任务状态、报告类型、市场类型使用枚举或受限字符串。
- 日期统一使用 ISO 格式字符串或 `datetime/date`，接口层明确转换。
- 股票代码在服务层统一规范化：
  - A 股：6 位数字。
  - 港股：5 位数字，不足左侧补 `0`。

### 5.2 错误处理

后端错误分为：

| 类型 | 示例 | 处理 |
| --- | --- | --- |
| 用户输入错误 | 股票代码格式错误 | 返回 `400`，前端展示 |
| 可恢复源站错误 | 超时、限流 | 重试、降速、记录日志 |
| 业务未命中 | 未找到报告 | 任务项失败，不中断批次 |
| 文件系统错误 | 保存目录不可写 | 阻止任务开始或任务项失败 |
| 程序错误 | 未捕获异常 | 记录完整日志，返回通用错误 |

要求：

- 批量任务中单个股票失败不得中断整个任务。
- 用户可见错误要讲人话，不直接暴露 Python traceback。
- 后端日志保留技术细节，便于定位问题。

### 5.3 文件下载规范

- 下载必须先写入临时文件，如 `.partial`。
- 下载完成后校验：
  - HTTP 状态码为 `200`。
  - 文件大小大于 `0`。
  - 年报 PDF 实际大小默认必须不小于 `1MB`；低于阈值时按疑似通知函、信函、非年报正文处理并跳过。
  - 文件头符合 PDF 基本特征，优先检查 `%PDF`。
- 校验通过后再原子重命名。
- 默认不覆盖已有文件。
- 文件名必须按 Windows/macOS 共同规则清洗非法字符。
- 文件路径必须使用 `pathlib.Path` 处理，不得用字符串拼接路径。
- 不得写死 `/Users/...`、`C:\...` 等平台专属路径。
- 默认保存目录：Windows 使用 `C:/reports`，macOS/Linux 使用用户下载目录下的 `reports` 文件夹。
- 文件名长度需要控制，避免 Windows 路径过长问题。

默认文件名：

```text
{market}_{code}_{name}_{year}_{report_type}_{announcement_date}.pdf
```

非法字符统一替换为 `_`：

```text
\ / : * ? " < > |
```

## 6. 报告匹配规范

报告匹配必须可测试、可解释。

### 6.1 匹配流程

1. 拉取候选公告。
2. 过滤取消、作废、明显无关公告。
3. 根据标题识别年份。
4. 根据标题识别报告类型。
5. 对正本、摘要、英文版、更正版打分。
6. 年报候选如果源站返回附件大小，则默认过滤小于 `1MB` 的候选，避免下载通知信函或非正文 PDF。
7. 选择最高分候选。
8. 分数低于阈值则返回“未找到”，不强行下载。

### 6.2 正本优先

优先级：

1. 更正后的完整报告。
2. 完整报告。
3. 英文版完整报告。
4. 摘要。

默认策略应跳过摘要，除非用户后续明确选择“允许下载摘要”。

### 6.3 港股特殊规则

- 港股年报优先用 `keyword=年报`。
- 港股半年报优先用 `keyword=中期报告`。
- 港股一季报、三季报允许返回未找到，不视为程序异常。
- 港股代码必须规范化为 5 位。

## 7. API 规范

### 7.1 命名

- REST 路径使用复数名词，如 `/api/tasks`、`/api/reports/search`。
- JSON 字段使用 snake_case，前后端统一。
- 任务 ID 使用 UUID 字符串。

### 7.2 响应格式

成功响应：

```json
{
  "data": {},
  "message": "ok"
}
```

列表响应：

```json
{
  "items": [],
  "total": 0
}
```

错误响应：

```json
{
  "error": {
    "code": "save_dir_not_writable",
    "message": "保存目录不可写"
  }
}
```

### 7.3 实时日志

- 优先使用 Server-Sent Events。
- 日志事件必须包含：`time`、`level`、`task_id`、`code`、`message`。
- 前端断线重连后，应能重新拉取任务当前状态。

## 8. 前端开发规范

### 8.1 UI 风格

延续截图方向：深色工具界面、荧光黄强调、紧凑布局。页面第一屏必须是工具，不做营销落地页。

要求：

- 控件状态清晰：空闲、下载中、成功、失败、跳过。
- 批量任务进行中时，关键参数不可随意改动。
- 请求间隔和并发数必须在界面可见。
- 源站限流或自动降速必须在状态栏或日志中可见。
- 不用大面积说明文字占据工具界面。

### 8.2 交互控件

- 报告类型使用 segmented control。
- 请求间隔使用 slider + 数字输入。
- 并发数使用 stepper。
- 保存目录使用路径输入 + 浏览按钮。
- 下载、暂停、继续、取消使用清晰按钮状态。
- 图标优先使用 lucide-react。

### 8.3 前端状态

- 前端不得自行判断下载成功，只能展示后端任务结果。
- 前端不得自己拼接 PDF URL。
- 任务状态以 `task_id` 为中心管理。
- 页面刷新后应能恢复最近任务状态，至少能看到任务结果。

## 9. 测试规范

### 9.0 测试优先流程

所有功能改动、缺陷修复、匹配规则调整、限速策略调整，都必须遵守测试优先流程：

1. 先写测试用例或更新现有测试用例。
2. 运行测试，确认新增/修改的测试在当前实现下失败。
3. 再改业务代码，让测试通过。
4. 补充必要的边界用例。
5. 最后运行相关测试集，确认没有回归。

例外情况：

- 纯文档修改不要求先写测试。
- 纯视觉微调可以不写自动化测试，但必须有手动验证记录。
- 探索性临时代码不能合入主分支；合入前仍要补测试。

禁止行为：

- 先实现功能，最后象征性补测试。
- 因测试难写而跳过核心逻辑测试。
- 为了让测试通过而降低断言质量。
- 删除失败测试但不修复对应问题。

### 9.1 后端测试

必须覆盖：

- A 股代码规范化。
- 港股代码补零。
- 报告标题年份识别。
- 报告类型匹配。
- 摘要跳过逻辑。
- 文件名非法字符清洗。
- 已存在文件跳过。
- 限速器最小间隔。
- 自动降速。
- 单个任务失败不影响批量任务。

### 9.2 集成测试

建议保留少量真实数据源测试，但不能频繁跑：

- `000001` A 股年报。
- `00700` 港股年报。
- 不存在股票代码。

真实源站测试必须遵守默认 `2.0 秒` 或更高请求间隔。

### 9.3 前端测试

至少覆盖：

- 表单校验。
- 批量列表添加/删除。
- 请求间隔输入边界。
- 任务日志渲染。
- 成功/失败/跳过结果展示。

### 9.4 测试记录

每次功能改动完成后，需要在提交说明、PR 描述或开发记录中写明：

```text
测试：
- 新增/更新了哪些测试用例
- 运行了哪些测试命令
- 是否有未覆盖风险
```

如果某项测试暂时无法自动化，必须写明手动验证步骤和原因。

## 10. 配置规范

配置来源优先级：

1. 命令行参数或环境变量。
2. 用户设置文件。
3. 默认配置。

建议配置项：

```text
DEFAULT_SAVE_DIR
DEFAULT_REQUEST_INTERVAL_SECONDS
MIN_REQUEST_INTERVAL_SECONDS
MAX_CONCURRENCY
REQUEST_TIMEOUT_SECONDS
MAX_RETRIES
MAX_BACKOFF_SECONDS
DATABASE_PATH
```

敏感信息不应写入仓库。当前项目不需要账号密码或 token。

## 10.1 跨平台运行规范

目标平台：

- macOS
- Windows 10/11

后端要求：

- 所有路径使用 `pathlib.Path`。
- 默认保存目录：Windows 使用 `C:/reports`，macOS/Linux 使用 `Path.home() / "Downloads" / "reports"`。
- 数据库路径、下载路径、临时文件路径都必须可通过配置覆盖。
- 文件写入前必须创建目录并检查可写性。
- 文件名清洗必须覆盖 Windows 禁止字符。
- 不依赖 shell 专属语法实现核心功能。

前端要求：

- UI 不展示写死的 macOS 示例路径作为默认值；示例路径只能出现在文档说明里。
- 保存目录输入允许 Windows 路径和 macOS 路径。
- 路径只作为字符串传给后端，前端不自行解析平台规则。
- 长路径在表格中要省略显示，完整路径可通过 tooltip 或复制功能查看。

启动脚本要求：

- macOS/Linux 提供 `start.sh`。
- Windows 提供 `start.bat`。
- 两个脚本都应完成前后端启动或给出清晰提示。
- 脚本不得包含用户个人绝对路径。

测试要求：

- 文件名清洗测试必须覆盖 Windows 禁止字符。
- 路径处理测试必须覆盖 Windows 风格路径和 POSIX 风格路径。
- CI 或手动验收中至少保留 Windows 启动说明检查项。

打包预留：

- MVP 先支持浏览器访问本地服务。
- 第二阶段预留 Tauri/Electron 桌面包装。
- 桌面包装不得绕过后端限速器和下载校验。

## 11. 日志规范

### 11.1 用户日志

用户日志用于 UI 展示，要求简洁：

```text
[21:30:12] 000001 找到 2024 年报：2024年年度报告
[21:30:14] 000001 下载完成：A股_000001_平安银行_2024_年报_2025-03-15.pdf
```

### 11.2 技术日志

技术日志用于排查问题，可包含：

- 请求 URL 的域名和路径。
- 状态码。
- 重试次数。
- 限速等待时间。
- 异常类型。

不记录大段 PDF 内容，不记录无关隐私路径。

## 12. Git 与提交规范

当前目录不是独立 Git 仓库时，先初始化项目仓库或确认仓库根目录，避免误提交用户主目录文件。

提交粒度：

- 一个提交只解决一个明确问题。
- 功能提交应包含对应测试；若没有测试，需要在提交说明中解释原因。
- 不把格式化全仓和功能改动混在一个提交。
- 不提交 `.venv`、`node_modules`、下载的 PDF、临时文件、数据库运行文件。

建议提交信息：

```text
feat: add report search api
feat: add rate-limited batch downloader
fix: skip existing pdf files safely
docs: add development standards
test: cover hk stock code normalization
```

## 13. 文档维护规范

以下变更必须同步更新文档：

- 新增或修改 API。
- 修改默认请求间隔、并发数、重试策略。
- 修改文件命名规则。
- 修改跨平台路径、启动脚本或打包策略。
- 修改报告匹配优先级。
- 新增外部数据源。
- 新增用户可见设置项。

## 14. 开发完成定义

一个功能只有同时满足以下条件才算完成：

1. 改动前已先写好对应测试用例，并确认测试能暴露当前缺口。
2. 功能可从 UI 或 API 正常使用。
3. 错误状态有用户可见反馈。
4. 批量任务不绕过限速器。
5. 不覆盖用户已有文件，除非用户明确选择。
6. 相关自动化测试已通过。
7. macOS 和 Windows 路径处理均符合跨平台规范。
8. 如无法自动化，已有清晰手动验证记录和原因说明。
9. 文档已同步更新。

## 15. MVP 实现优先级

第一优先级：

- 独立 Python 环境。
- 单股报告搜索。
- PDF URL 解析。
- 全局限速器。
- 单股下载。
- 类截图 UI。

第二优先级：

- 批量手动输入。
- 请求间隔与并发数 UI。
- 实时日志。
- 失败重试。

第三优先级：

- Excel/CSV 导入。
- SQLite 任务历史。
- 年份范围批量匹配。
- 桌面打包。
